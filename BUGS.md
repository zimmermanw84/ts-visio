# ts-visio Bug Report

Generated by static analysis of the codebase. Bugs are grouped by severity.

---

## HIGH

### Bug 21: `SwimlanePattern.addLane` never attaches lanes to the pool

**File:** `src/diagrams/SwimlanePattern.ts` line 17

`addPool` creates a vertical `List`. `addLane` creates a plain `Container`. Neither method adds the lane as a `List` member of the pool — the user gets two completely independent shapes. The swimlane API is effectively non-functional. `page.addSwimlanePool()` / `page.addSwimlaneLane()` produce no connected structure.

**Fix direction:** `addLane` must accept the pool shape and call `pool.addListItem(lane)`.

## MEDIUM

### Bug 22: `MasterManager.addMasterEntry` writes redundant inline `<Shapes>` into `masters.xml`

**File:** `src/core/MasterManager.ts` lines 327–344

The canonical master geometry lives in the linked `masterN.xml` (`MasterContents`). Real Visio `masters.xml` Master entries contain only `<PageSheet>` and `<Rel>` — the inline `<Shapes>` is a legacy/redundant element. Writing it here means both files carry the shape data and they can silently diverge after edits. Visio prefers the linked file, but the duplication inflates `masters.xml` and is non-standard.

**Fix direction:** Remove `Shapes: { Shape: masterShape }` from the `addMasterEntry` call; the shape content is already written to `masterN.xml`.

### Bug 23: `MasterManager.importFromStencil` O(n²) — calls `this.load()` inside loop

**File:** `src/core/MasterManager.ts` line 154

Each iteration re-reads and re-parses `masters.xml` from disk to find the current max ID. With a 50-master stencil that's 50 full parse cycles. The ID could be tracked with a simple counter initialized once before the loop.

**Fix direction:** Capture the initial existing IDs before the loop, then increment a local counter each iteration.

---

### Bug 24: `Page.getShapes()` / `getShapeById()` / `getConnectors()` return stale data when `autoSave=false`

**File:** `src/Page.ts` lines 115–207

All read methods create `new ShapeReader(this.pkg)` and call `pkg.getFileText(pagePath)`. When `autoSave=false`, mutations are held only in `PageXmlCache.dirtyPages` and never written to `pkg` until `flush()`. Every read after a write in `autoSave=false` mode returns the pre-mutation state. `ShapeModifier.getShapeChildren` (line 872) has the same problem.

**Fix direction:** Either document this clearly or have the read methods consult `PageXmlCache` directly when pages are dirty.

### Bug 26: `getPageLayers` return type omits the `print` property

**File:** `src/core/LayerEditor.ts` line 126

`addLayer` accepts and persists `options.print` as `<Cell N="Print" V="0|1"/>`, but `getPageLayers` only reads `Name`, `Visible`, and `Lock`. The `print` field is silently dropped on read-back, so round-tripping a layer's properties through the API discards the print setting.

**Fix direction:** Add `print: getVal('Print') !== '0'` to the returned object and extend the return type.

### Bug 28: `ConnectorEditor.addConnector` unconditionally adds a MASTERS relationship

**File:** `src/core/ConnectorEditor.ts` line 62

The generated connector shape has no `@_Master` attribute, so the page→`masters.xml` relationship is unnecessary. It is idempotent and harmless, but it pollutes every page that ever receives a connector with an unneeded OPC relationship.

**Fix direction:** Only add the MASTERS relationship when the connector explicitly references a master shape.

### Bug 29: `RelsManager.ensureRelationship` fragile relationship ID parsing

**File:** `src/core/RelsManager.ts` line 56

```ts
const idNum = parseInt(r['@_Id'].replace('rId', ''));
```

Assumes all relationship IDs use the `rId` prefix. IDs with other formats (e.g. `R1`, `rid1`, a bare integer, or `undefined`) cause `parseInt` to return `NaN`, leaving `maxId = 0`, so the new ID becomes `rId1` — which may duplicate an existing relationship.

**Fix direction:** Use a regex to extract the numeric suffix regardless of prefix, or fall back gracefully when no numeric suffix is found.
