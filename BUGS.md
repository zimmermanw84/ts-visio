# ts-visio Bug Report

Generated by static analysis of the codebase. Bugs are grouped by severity.

---

## HIGH

### Bug 1: Hardcoded page path bypasses `pagePathRegistry`

**File:** `src/core/ShapeModifier.ts` (multiple locations)

Multiple locations construct the page XML path as `` `visio/pages/page${pageId}.xml` `` instead of looking it up from `pagePathRegistry`. This breaks files where Visio used non-sequential page filenames (e.g. `page3.xml` for the first page after deletions).

**Fix direction:** Always resolve page paths via `pagePathRegistry.get(pageId)` and throw if the entry is missing.

---

### Bug 7: `updateShapeDimensions` does not update LocPinX/LocPinY

**File:** `src/core/ShapeModifier.ts`

When resizing a shape, the local pin (rotation center) is not updated proportionally. For shapes that have been moved or rotated this causes visible drift each time width/height is changed.

**Fix direction:** After updating `Width`/`Height`, recompute `LocPinX = Width / 2` and `LocPinY = Height / 2` (or preserve the existing ratio if the shape uses a custom pin).

---

### Bug 18/19: Multiple independent `ShapeModifier`/`PageXmlCache` instances per document

**File:** `src/VisioDocument.ts` lines 66–100, `src/Page.ts` line 49

`VisioDocument.pages` creates a fresh `Page` (and thus a fresh `ShapeModifier` + `PageXmlCache`) every time `_pageCache` is invalidated (after `addPage`, `deletePage`, `movePage`, etc.). Two `Page` references to the same page each maintain their own XML cache — whoever writes last silently wins and the other's mutations are lost.

**Fix direction:** Construct a single `ShapeModifier` on `VisioDocument` and pass it to every `Page` instance. The `Page` constructor already accepts an optional `modifier` parameter but `VisioDocument` never passes one.

---

### Bug 20: Section replacement destroys non-library cells on style updates

**File:** `src/core/ShapeModifier.ts` (style update methods)

`updateShapeStyle` (e.g. setting `fillColor`) filters out the entire `Fill`/`Line`/`Character` section and replaces it with a freshly-built one. Any extra cells present in those sections from real Visio files (`FillForegndTrans`, `ShdwPattern`, etc.) are permanently lost on the first style mutation.

**Fix direction:** Instead of replacing the entire section, upsert individual cells within the existing section. Only create a new section if one does not already exist.

---

## MEDIUM

### Bug 3: Unwanted Line section added when only `fillColor` is set

**File:** `src/core/ShapeModifier.ts`

Setting only `fillColor` on a shape writes both a `Fill` and a `Line` section, causing unexpected line styling to appear on shapes that should have none.

**Fix direction:** Only emit the `Line` section when at least one line property (`lineColor`, `lineWeight`, `linePattern`) is explicitly set.

---

### Bug 4: Connection point `@_V` always `'0'`

**File:** `src/core/ConnectorEditor.ts` (or connection point builder)

All generated connection point entries have `V="0"`, which disables the point in non-formula-aware renderers. The value should be the actual coordinate.

**Fix direction:** Populate `V` with the evaluated coordinate value alongside the formula in `F`.

---

### Bug 5: `resizeShape` only matches cells with exact formula strings `'Width'`/`'Height'`

**File:** `src/core/ShapeModifier.ts`

Real Visio shape cells often use references like `Width*0.5` or a raw numeric value. The current exact-match approach misses these, leaving geometry cells stale after a resize.

**Fix direction:** Match by cell `N` attribute (name) rather than by formula value when updating geometry references.

---

### Bug 8: Foreign shapes missing image dimension cells

**File:** `src/core/ShapeModifier.ts` (foreign/image shape creation)

Image shapes are created without `ImgWidth`/`ImgHeight` cells, which can cause display issues in Visio Desktop.

**Fix direction:** Emit `ImgWidth` and `ImgHeight` cells (matching `Width`/`Height`) when creating Foreign shapes.

---

### Bug 12: `deleteShape` on groups leaves orphaned `Connect` entries for child shapes

**File:** `src/core/ShapeModifier.ts` (deleteShape logic)

When a group is deleted, only the top-level shape's connectors are cleaned up. Connectors attached to child shapes inside the group leave dangling `<Connect>` entries referencing non-existent shape IDs.

**Fix direction:** When deleting a shape, recursively collect all descendant shape IDs (via `getChildren`) and remove all `<Connect>` elements that reference any of them.

---

## LOW

### Bug 2: Page ID string/number type inconsistencies

**File:** `src/core/PageManager.ts`, `src/core/ShapeModifier.ts`

Page IDs are sometimes compared as strings and sometimes as numbers (`==` vs `===`, `parseInt` coercions). While currently benign, this is fragile.

**Fix direction:** Standardize page IDs as strings throughout and use strict equality everywhere.

---

### Bug 6: `PageXmlCache` stale `content` field

**File:** `src/core/PageXmlCache.ts`

The `content` field can become stale after mutations but the cache re-parses from the raw buffer on each access, so it works by accident. This is a latent correctness issue if the access pattern changes.

**Fix direction:** Invalidate or regenerate `content` explicitly after every write.

---

### Bug 9: Generated `Geometry` sections missing `NoShow` cell

**File:** `src/utils/GeometryBuilder.ts`

The Visio spec requires a `NoShow` cell in every `Geometry` section. Omitting it may cause rendering issues in strict Visio implementations.

**Fix direction:** Add `<Cell N="NoShow" V="0"/>` to each generated `Geometry` section.

---

### Bug 10: Connectors missing style attribute references

**File:** `src/core/ConnectorEditor.ts`

Generated connector shapes are missing `LineStyle`, `FillStyle`, and `TextStyle` attribute references that Visio Desktop expects.

**Fix direction:** Add default style references (`LineStyle="0"`, `FillStyle="0"`, `TextStyle="0"`) to connector shape elements.

---

### Bug 14: Initial `page1.xml` template has no `<PageSheet>` element

**File:** `src/templates/MinimalVsdx.ts`

The `PAGE1_XML` template contains only `<Shapes/>` with no `<PageSheet>`. Pages added via `addPage()` do include a full PageSheet. This inconsistency means the initial page may be treated differently by Visio Desktop and the assumed 8.5×11 default is not written to disk.

**Fix direction:** Add a `<PageSheet>` with standard US Letter dimensions and a `<Connects/>` element to `PAGE1_XML`.

---

### Bug 15: Loose `==` vs `===` for shape ID comparisons

**File:** `src/core/ContainerEditor.ts` line ~105

`shapes.findIndex((s: any) => s['@_ID'] == shapeId)` uses `==` while all other ID comparisons use `===`.

**Fix direction:** Use strict equality `===` consistently.

---

### Bug 16: `deleteLayer` leaves index gaps in remaining layers

**File:** `src/core/LayerEditor.ts`

When a layer is deleted, remaining layers keep their original IX values. Subsequent `addLayer` calls find the highest existing index and continue from there, leaving a permanent gap. While Visio handles gaps, it is unintuitive.

**Fix direction:** Either re-index remaining layers after deletion (and update all `LayerMember` cell values on shapes), or document that layer indices are stable non-contiguous identifiers.

---

### Bug 17: `getConnectors()` silently drops partially-connected connectors

**File:** `src/ShapeReader.ts` line ~120

`if (!entry?.beginConnect || !entry?.endConnect) continue;` skips connectors with only one endpoint connected. The user receives no indication that connectors were omitted.

**Fix direction:** Either return partial connectors with `undefined` for the missing endpoint, or emit a warning when connectors are skipped.
