# ts-visio Bug Report

Generated by static analysis of the codebase. Bugs are grouped by severity.



---

## HIGH

### ~~Bug 1: Hardcoded page path bypasses `pagePathRegistry`~~ ✅ Fixed in v1.16.2

**File:** `src/ShapeModifier.ts` line 266

`ensureRelationship` for master shapes was called with a hardcoded `` `visio/pages/page${pageId}.xml` `` path instead of `this.cache.getPagePath(pageId)`. In files where Visio used non-sequential page filenames (e.g. `page3.xml` for the first page after deletions), this wrote the relationship to the wrong `.rels` file.

**Fix:** Replaced the hardcoded template string with `this.cache.getPagePath(pageId)`, consistent with all other path lookups in `ShapeModifier`.

---

### ~~Bug 7: `updateShapeDimensions` does not update LocPinX/LocPinY~~ ✅ Fixed in v1.16.3

**File:** `src/core/PageXmlCache.ts`

`updateShapeDimensions` only wrote `Width` and `Height`, leaving `LocPinX`/`LocPinY` stale. For rotated or off-centre shapes this caused visible drift on every dimension change. The bug affected both the public `ShapeModifier.updateShapeDimensions` path and `ContainerEditor.resizeContainerToFit`.

**Fix:** Added `upsert('LocPinX', (w / 2).toString())` and `upsert('LocPinY', (h / 2).toString())` to `PageXmlCache.updateShapeDimensions`, matching the pattern already used in `resizeShape`. Two regression tests added to `RotationResize.test.ts`.

---

### ~~Bug 18/19: Multiple independent `ShapeModifier`/`PageXmlCache` instances per document~~ ✅ Fixed in v1.16.1

**File:** `src/VisioDocument.ts` lines 66–100, `src/Page.ts` line 49

`VisioDocument.pages` created a fresh `Page` (and thus a fresh `ShapeModifier` + `PageXmlCache`) every time `_pageCache` was invalidated. Two `Page` references to the same page each maintained their own XML cache — whoever wrote last silently won.

**Fix:** `VisioDocument` now owns a single `ShapeModifier` instance (initialized in the constructor) and passes it to every `new Page(...)` call via the existing optional `modifier` parameter.

---

### ~~Bug 20: Section replacement destroys non-library cells on style updates~~ ✅ Fixed in v1.16.4

**File:** `src/core/ShapeModifier.ts` (style update methods)

`updateShapeStyle` (e.g. setting `fillColor`) filters out the entire `Fill`/`Line`/`Character` section and replaces it with a freshly-built one. Any extra cells present in those sections from real Visio files (`FillForegndTrans`, `ShdwPattern`, etc.) are permanently lost on the first style mutation.

**Fix:** Instead of replacing the entire section, each style property now upserts only the relevant cell(s) within the existing section. A new section is created only when none exists. Character style bits (bold/italic/underline/strikethrough) are merged with existing values rather than recalculated from scratch. Five regression tests added to `StyleUpsert.test.ts`.

## MEDIUM

### ~~Bug 3: Unwanted Line section added when only `fillColor` is set~~ ✅ Fixed in v1.16.5

**File:** `src/shapes/ShapeBuilder.ts`

Setting only `fillColor` on a shape writes both a `Fill` and a `Line` section, causing unexpected line styling to appear on shapes that should have none.

**Fix:** Changed the condition in `ShapeBuilder.createStandardShape` from `if (props.fillColor || hasLineProps)` to `if (hasLineProps)`, so the Line section is only emitted when at least one line property (`lineColor`, `linePattern`) is explicitly set. Four regression tests added to `Styling.test.ts`.

---

### ~~Bug 4: Connection point `@_V` always `'0'`~~ ✅ Fixed in v1.16.6

**File:** `src/shapes/ConnectionPointBuilder.ts`

All generated connection point entries have `V="0"`, which disables the point in non-formula-aware renderers. The value should be the actual coordinate.

**Fix:** `buildRow` and `buildConnectionSection` now accept optional `width`/`height` parameters and compute `@_V` as `width * xFraction` / `height * yFraction`. `ShapeBuilder` passes `props.width`/`props.height`; `ShapeModifier.addConnectionPoint` reads the shape's `Width`/`Height` cells. Three regression tests added to `ConnectionPoints.test.ts`.

---

### ~~Bug 5: `resizeShape` only matches cells with exact formula strings `'Width'`/`'Height'`~~ ✅ Fixed in v1.16.7

**File:** `src/core/ShapeModifier.ts`

Real Visio shape cells often use references like `Width*0.5` or a raw numeric value. The current exact-match approach misses these, leaving geometry cells stale after a resize.

**Fix:** Replaced the exact `===` checks with a formula-evaluation approach: any geometry cell formula containing `Width` or `Height` has those identifiers substituted with the new numeric values, then the resulting arithmetic expression is safely evaluated (validated against `/^[\d\s.+\-*/()]+$/` before `Function()` invocation) and written to `@_V`. Three regression tests added to `RotationResize.test.ts`.

---

### ~~Bug 8: Foreign shapes missing image dimension cells~~ ✅ Fixed in v1.16.8

**File:** `src/shapes/ForeignShapeBuilder.ts`

Image shapes are created without `ImgWidth`/`ImgHeight` cells, which can cause display issues in Visio Desktop.

**Fix:** Added `ImgWidth` and `ImgHeight` cells to `ForeignShapeBuilder.createImageShapeObject`, with `@_V` matching `Width`/`Height` and `@_F` set to `'Width'`/`'Height'`. Regression test added to `ImageShapeStructure.test.ts`.

---

### ~~Bug 12: `deleteShape` on groups leaves orphaned `Connect` entries for child shapes~~ ✅ Fixed in v1.16.9

**File:** `src/core/ShapeModifier.ts` (deleteShape logic)

When a group is deleted, only the top-level shape's connectors are cleaned up. Connectors attached to child shapes inside the group leave dangling `<Connect>` entries referencing non-existent shape IDs.

**Fix:** Added `findShapeInTree` and `collectShapeIds` private helpers to `ShapeModifier`. `deleteShape` now collects all descendant IDs before removal and filters both `Connect` and `Relationship` entries using that full set. Regression test added to `DeleteShape.test.ts`.

## LOW

### ~~Bug 2: Page ID string/number type inconsistencies~~ ✅ Fixed in v1.16.10

**File:** `src/core/PageManager.ts`, `src/core/ShapeModifier.ts`

Page IDs are sometimes compared as strings and sometimes as numbers (`==` vs `===`, `parseInt` coercions). While currently benign, this is fragile.

**Fix:** Changed `PageEntry.id` from `number` to `string`. Replaced `parseInt(node['@_ID'])` with `String(node['@_ID'])`. Numeric comparisons in `createPageInternal` and `duplicatePage` now use `Number(p.id)` locally. All `p.id.toString() === pageId` comparisons replaced with direct strict `p.id === pageId`. Updated `VisioDocument` to drop the redundant `.toString()` call. Two regression tests added to `PageManager.test.ts`.

---

### ~~Bug 6: `PageXmlCache` stale `content` field~~ ✅ Fixed in v1.16.11

**File:** `src/core/PageXmlCache.ts`

The `content` field can become stale after mutations but the cache re-parses from the raw buffer on each access, so it works by accident. This is a latent correctness issue if the access pattern changes.

**Fix:** Added a dirty-page guard to `getParsed`: when a page is in `dirtyPages` (has unflushed mutations), the cached `parsed` object is returned immediately without comparing against `pkg.getFileText()`. This prevents external writes to `pkg` between mutations from causing a cache miss that would re-parse stale content and discard in-flight mutations. Regression test added to `ShapeModifier_AutoSave.test.ts`.

---

### Bug 9: Generated `Geometry` sections missing `NoShow` cell

**File:** `src/utils/GeometryBuilder.ts`

The Visio spec requires a `NoShow` cell in every `Geometry` section. Omitting it may cause rendering issues in strict Visio implementations.

**Fix direction:** Add `<Cell N="NoShow" V="0"/>` to each generated `Geometry` section.

---

### Bug 10: Connectors missing style attribute references

**File:** `src/core/ConnectorEditor.ts`

Generated connector shapes are missing `LineStyle`, `FillStyle`, and `TextStyle` attribute references that Visio Desktop expects.

**Fix direction:** Add default style references (`LineStyle="0"`, `FillStyle="0"`, `TextStyle="0"`) to connector shape elements.

---

### Bug 14: Initial `page1.xml` template has no `<PageSheet>` element

**File:** `src/templates/MinimalVsdx.ts`

The `PAGE1_XML` template contains only `<Shapes/>` with no `<PageSheet>`. Pages added via `addPage()` do include a full PageSheet. This inconsistency means the initial page may be treated differently by Visio Desktop and the assumed 8.5×11 default is not written to disk.

**Fix direction:** Add a `<PageSheet>` with standard US Letter dimensions and a `<Connects/>` element to `PAGE1_XML`.

---

### Bug 15: Loose `==` vs `===` for shape ID comparisons

**File:** `src/core/ContainerEditor.ts` line ~105

`shapes.findIndex((s: any) => s['@_ID'] == shapeId)` uses `==` while all other ID comparisons use `===`.

**Fix direction:** Use strict equality `===` consistently.

---

### Bug 16: `deleteLayer` leaves index gaps in remaining layers

**File:** `src/core/LayerEditor.ts`

When a layer is deleted, remaining layers keep their original IX values. Subsequent `addLayer` calls find the highest existing index and continue from there, leaving a permanent gap. While Visio handles gaps, it is unintuitive.

**Fix direction:** Either re-index remaining layers after deletion (and update all `LayerMember` cell values on shapes), or document that layer indices are stable non-contiguous identifiers.

---

### Bug 17: `getConnectors()` silently drops partially-connected connectors

**File:** `src/ShapeReader.ts` line ~120

`if (!entry?.beginConnect || !entry?.endConnect) continue;` skips connectors with only one endpoint connected. The user receives no indication that connectors were omitted.

**Fix direction:** Either return partial connectors with `undefined` for the missing endpoint, or emit a warning when connectors are skipped.
